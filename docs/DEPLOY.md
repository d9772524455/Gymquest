# DEPLOY.md — Gym Quest production runbook

Runbook for deploying, maintaining, and troubleshooting Gym Quest on reg.ru Cloud.
Audience: ops / maintainer. Sibling docs: `README.md` (onboarding), `CLAUDE.md` (AI
assistant brief), `docs/audit-report.md` (open issues — this doc closes **I2**).

---

## 1. Prod topology

- **Host:** reg.ru Cloud VDS (Ubuntu 20.04, 2 vCPU, 2 GB RAM, 20 GB disk). One host
  does everything — nginx, app container, Postgres container.
- **Domain:** `gymquest.ru` — `A` record points to the VDS IP (e.g. `194.67.102.76`).
  `www.gymquest.ru` is a second `A` record with the same target. Any stale `AAAA`
  records (from the old reg.ru shared hosting) must be removed.
- **Reverse proxy:** nginx listens on `:80` and `:443`, upstream is
  `http://127.0.0.1:3000` (the `app` container binds to loopback only — see
  `docker-compose.yml`).
- **App runtime:** `docker compose up -d` brings up two services:
  - `db` — `postgres:16-alpine`, bound to `127.0.0.1:5432`, SSL enabled via a
    self-signed cert bind-mounted from `./db-ssl/` (because the app forces SSL in
    `NODE_ENV=production`).
  - `app` — Node 22 server built from the repo `Dockerfile`. Exposes `:3000` on
    loopback, has a built-in `HEALTHCHECK` that hits `/api/health` every 30 s.
- **CI/CD:** `.github/workflows/deploy.yml` runs on every push to `main`. It opens
  an SSH session to the VDS (via `appleboy/ssh-action`) and runs
  `git pull origin main && docker compose up -d --build` inside `/opt/gymquest`.

---

## 2. Prerequisites (first-time VDS setup)

- Ubuntu 20.04 or newer.
- Docker Engine 24+ and Docker Compose v2 (`docker compose` — the plugin form, not
  the legacy `docker-compose` binary).
- `nginx` installed via apt (`sudo apt install nginx`).
- `git`.
- `certbot` + nginx plugin: `sudo apt install certbot python3-certbot-nginx`.
- An SSH public key added to `/root/.ssh/authorized_keys` for the maintainer. The
  same key pair is used by GitHub Actions (see §7).

## 3. First-time deploy (bootstrap from scratch)

Run as `root` on the VDS.

```bash
# 1. Clone the repo into /opt/gymquest
cd /opt
git clone https://github.com/d9772524455/Gymquest gymquest
cd /opt/gymquest

# 2. Create the compose env file from the template
cp .env.example .env

# 3. Generate strong secrets and edit .env
#    POSTGRES_PASSWORD — any strong random string
#    JWT_SECRET        — exactly 64 random hex chars
openssl rand -hex 32                      # copy output into JWT_SECRET
openssl rand -hex 24                      # copy output into POSTGRES_PASSWORD
nano .env
chmod 600 .env                            # keep secrets off other local users

# 4. Generate self-signed Postgres SSL cert (one-off, cert is valid 10 years)
bash scripts/generate-db-ssl.sh
# Produces db-ssl/server.crt and db-ssl/server.key — both gitignored.

# 5. Build and start the stack
docker compose up -d --build

# 6. Wait for healthcheck and verify locally
docker compose ps                         # both services should be "healthy"
curl -s http://localhost:3000/api/health  # → {"status":"ok",...}
```

If `/api/health` returns JSON with `"status":"ok"`, the app stack is running. Next:
wire up nginx and SSL (§4 and §5).

## 4. Nginx base config (port 80)

The repo ships a reference config at `nginx.conf`. Install it on the VDS:

```bash
# On VDS
sudo cp /opt/gymquest/nginx.conf /etc/nginx/sites-available/gymquest.ru
sudo ln -sf ../sites-available/gymquest.ru /etc/nginx/sites-enabled/gymquest.ru
sudo rm -f /etc/nginx/sites-enabled/default

sudo nginx -t
sudo systemctl reload nginx
```

At this point `http://gymquest.ru` should proxy to the app container. The port-80
block already carries `X-Frame-Options`, `X-Content-Type-Options`, and
`X-XSS-Protection`. The full header set goes onto the 443 block in §6.

---

## 5. Certbot — provision HTTPS

certbot's nginx plugin reads the existing port-80 server block, writes a new
`server { listen 443 ssl; ... }` block into the same file, and configures the
`:80 → :443` redirect automatically.

```bash
sudo certbot --nginx --non-interactive --agree-tos --redirect \
  --email you@domain.tld -d gymquest.ru -d www.gymquest.ru
```

After this succeeds, open `/etc/nginx/sites-enabled/gymquest.ru` — it should now
contain both a port-80 block (with the redirect) and a port-443 block managed by
certbot. Proceed to §6 to harden the 443 block.

---

## 6. Canonical 443 block with security headers — closes I2

**Why this section exists.** Audit issue I2 calls for nginx 443 to mirror the full
security-header set on HTTPS. certbot only adds SSL directives — it does not
touch the `add_header` lines from the port-80 block. The template below is the
source of truth for what the 443 block on the VDS must look like.

Edit `/etc/nginx/sites-enabled/gymquest.ru`. The 443 block (already partially
generated by certbot) should end up looking exactly like this:

```nginx
server {
    listen 443 ssl http2;
    listen [::]:443 ssl http2;
    server_name gymquest.ru www.gymquest.ru;

    # SSL — managed by certbot
    ssl_certificate /etc/letsencrypt/live/gymquest.ru/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/gymquest.ru/privkey.pem;
    include /etc/letsencrypt/options-ssl-nginx.conf;
    ssl_dhparam /etc/letsencrypt/ssl-dhparams.pem;

    client_max_body_size 5M;

    gzip on;
    gzip_types text/plain text/css application/json application/javascript text/xml application/xml application/xml+rss;
    gzip_min_length 256;

    # Security headers (closes audit I2 — mirror on 443)
    add_header X-Frame-Options DENY always;
    add_header X-Content-Type-Options nosniff always;
    add_header X-XSS-Protection "1; mode=block" always;
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
    add_header Referrer-Policy "strict-origin-when-cross-origin" always;
    add_header Content-Security-Policy "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline' https://fonts.googleapis.com; font-src 'self' https://fonts.gstatic.com; img-src 'self' data:; connect-src 'self'" always;

    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_cache_bypass $http_upgrade;
        proxy_read_timeout 90s;
    }

    location /app/ {
        proxy_pass http://127.0.0.1:3000/app/;
        proxy_set_header Host $host;
        proxy_set_header X-Forwarded-Proto $scheme;
        expires 1h;
        add_header Cache-Control "public";
    }

    location /dashboard/ {
        proxy_pass http://127.0.0.1:3000/dashboard/;
        proxy_set_header Host $host;
        proxy_set_header X-Forwarded-Proto $scheme;
        expires 1h;
        add_header Cache-Control "public";
    }
}
```

**Notes on the CSP.**

- `script-src 'self'` is strict — Phase 2 removed all inline `<script>` from the
  client and dashboard HTML, so no `'unsafe-inline'` is needed for JS.
- `style-src 'self' 'unsafe-inline'` is intentional. The client/dashboard HTML
  still carries inline `style=""` attributes (spec §4.4 left inline styles out of
  scope). Dropping `'unsafe-inline'` here would break rendering — do not remove
  it without a follow-up refactor.
- `https://fonts.googleapis.com` (stylesheets) and `https://fonts.gstatic.com`
  (font files) allow Google Fonts, which the frontend uses.
- `img-src 'self' data:` covers inlined hero avatars and QR PNGs.
- `connect-src 'self'` keeps all `fetch`/`XHR` traffic on-origin — the API is
  same-origin, so no third-party hosts are needed.

Apply, validate, and verify:

```bash
sudo nano /etc/nginx/sites-enabled/gymquest.ru   # merge the template above
sudo nginx -t
sudo systemctl reload nginx

# Verify the headers are live on HTTPS
curl -sI https://gymquest.ru | grep -iE 'strict-transport|x-frame|content-security|referrer-policy|x-content-type'
```

You should see one line per header in the output.

## 7. CI/CD pipeline

Two workflows live under `.github/workflows/`:

- **`ci.yml`** — runs on every PR and on push to `main`. Three jobs:
  - `lint-typecheck` — `npm run lint` + `cd server && npm run typecheck`.
  - `unit` — `cd server && npm test` (server unit tests).
  - `smoke` — spins up a Postgres service, runs migrations, starts the API, and
    runs `npm run smoke` against `http://localhost:3000`.
    All three must go green before a PR is merged.

- **`deploy.yml`** — runs on push to `main`. SSHes to the VDS and executes
  `cd /opt/gymquest && git pull origin main && docker compose up -d --build`.

**Required GitHub Secrets** (`Settings → Secrets and variables → Actions`):

| Secret        | Value                                                                |
| ------------- | -------------------------------------------------------------------- |
| `VPS_HOST`    | VDS IP (e.g. `194.67.102.76`)                                        |
| `VPS_USER`    | `root`                                                               |
| `VPS_SSH_KEY` | ed25519 **private** key; pubkey goes in `/root/.ssh/authorized_keys` |

The deploy key is a dedicated ed25519 pair generated for this workflow — do not
reuse personal SSH keys. The public half lives on the VDS; the private half only
exists as a GitHub secret.

## 8. Post-deploy verify

Run from a local machine (not from the VDS). This hits the real domain over HTTPS:

```bash
TARGET=prod node scripts/smoke.mjs
# Expected final line: "All smoke checks passed"
```

The smoke script runs 8 happy-path checks (register club, register member, login,
workout checkin, leaderboard, QR token, QR checkin, cleanup). If any step fails,
it exits non-zero and logs the failing step.

---

## 9. SSL renewal

certbot installs a systemd timer that runs twice daily and renews certificates
within 30 days of expiry.

```bash
# Confirm the timer is active
systemctl list-timers | grep certbot

# Dry-run — safe, does not write anything
sudo certbot renew --dry-run

# Force a real renewal (only if you know why)
sudo certbot renew
```

After a successful renewal certbot reloads nginx automatically via its deploy
hook. If the hook breaks, nginx keeps serving the old cert until the next manual
reload.

## 10. Rollback

If a deploy introduces a regression, revert on the VDS first (fast fix), then
revert on `main` so the next deploy does not re-apply the broken commit.

```bash
# 1. On VDS — find the last known-good SHA
ssh root@194.67.102.76 "cd /opt/gymquest && git log --oneline -10"

# 2. Reset the VDS checkout to that SHA and rebuild
ssh root@194.67.102.76 "cd /opt/gymquest && git reset --hard <SHA> && docker compose up -d --build"

# 3. On local — revert the bad merge in main so CI/CD does not re-push it
git checkout main
git pull
git revert -m 1 <bad-merge-SHA>
git push origin main
```

> Destructive commands (`git reset --hard`, `git revert` on `main`) should only
> be run after confirming with the owner. A forced rollback without agreement can
> lose work.

## 11. SMTP note (known blocker)

reg.ru Cloud blocks outbound SMTP on ports **25 / 465 / 587 / 2525** by default.
The app's email alert flow (`POST /api/club/send-alerts`, via nodemailer) will
time out trying to reach `smtp.gmail.com:587` even when `SMTP_*` env vars are set
correctly.

This is a reg.ru account-level policy, not a bug. The account owner (Дмитрий)
must open a support ticket from the reg.ru panel and request SMTP egress
unblocking. Until then, the `/api/club/send-alerts` endpoint returns successfully
but sends zero emails.

Workaround, if needed urgently: swap nodemailer for a transactional-email HTTP
API (SendGrid, Mailgun, Postmark) — but that is a code change, not a deploy fix.

## 13. Troubleshooting

| Symptom                                       | Likely cause                                | Where to look                                                                                                                                                                   |
| --------------------------------------------- | ------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `502 Bad Gateway` on HTTPS                    | app container down or not listening         | `docker compose ps`; `docker compose logs app`. Usual root cause: `JWT_SECRET` or `DATABASE_URL` missing — `server/src/config.js` fail-fasts on startup.                        |
| Migrations fail on container start            | `db` not healthy, or SSL cert files missing | `docker compose logs db`. Check `db-ssl/server.crt` and `db-ssl/server.key` exist and are readable. Re-run `bash scripts/generate-db-ssl.sh` if needed.                         |
| CI smoke fails but local smoke passes         | GitHub Actions secrets missing or wrong     | `Settings → Secrets and variables → Actions`. `VPS_*` set? For the smoke job itself, no secrets are needed — it runs against an in-job Postgres service.                        |
| Certificate expiring within days              | auto-renew hook broken                      | `sudo certbot renew --dry-run`. If the dry-run fails, inspect `/etc/letsencrypt/renewal/gymquest.ru.conf` — especially the `renew_hook` line — and re-run `sudo certbot renew`. |
| Deploy workflow times out / red               | SSH auth failure or VDS unreachable         | GitHub Actions run log → `Deploy to VPS` step. Verify `VPS_HOST` reachable, `VPS_SSH_KEY` matches a line in `/root/.ssh/authorized_keys` on the VDS, `VPS_USER=root`.           |
| `curl https://gymquest.ru -I` missing headers | 443 block in nginx not updated              | Re-read §6 and diff `/etc/nginx/sites-enabled/gymquest.ru` against the canonical template. `sudo nginx -t && sudo systemctl reload nginx`.                                      |
| DNS resolves to the wrong IP                  | stale `AAAA` record from old shared hosting | reg.ru DNS panel → delete the `AAAA` record. Keep only the `A` records pointing at the VDS.                                                                                     |

## 14. Updating the APK

The APK served at `/download` lives on the VDS at `/opt/gymquest/public/gymquest.apk`. It is **not** in git and **not** in the Docker image — it's bind-mounted read-only into the `app` container (see `docker-compose.yml`).

### Building a new APK

```bash
cd mobile
eas build --platform android --profile preview
```

Wait for the build, then download the `.apk` artefact locally (EAS gives you a URL to the artefact on completion).

### Uploading

```bash
# From repo root
scripts/upload-apk.sh ~/Downloads/build-xxx.apk
```

The script uses `root@194.67.102.76` by default. Pass a different target as the second arg if needed.

Upload is atomic: the APK goes to `gymquest.apk.new` first and is then renamed into place via `mv` on the same filesystem. In-flight `/download` streams keep serving the old file descriptor cleanly.

### Verifying

```bash
curl -I https://gymquest.ru/download
# Expect: HTTP/1.1 200 OK
#         Content-Disposition: attachment; filename="gymquest.apk"
#         Content-Type: application/vnd.android.package-archive
```

When no APK is present, `/download` returns `503 { error: { code: "APK_UNAVAILABLE" } }` — that's the expected state in CI and dev.
